> What Libretto does to reduce detection risk, and what to watch out for.

<Warning>
  Bot detection avoidance is best-effort. Libretto cannot guarantee your
  automation will go undetected. Using Libretto with authenticated accounts may
  violate the terms of service of those services. Understand the risks before
  automating against any site you don't control.
</Warning>

### What Libretto does automatically

Libretto applies two baseline stealth measures when launching a browser:

- **Suppresses the automation flag** — Disables Chrome's `AutomationControlled` blink feature so `navigator.webdriver` is not set to `true`.
- **Sets a real User-Agent** — Overrides the default Playwright user-agent with a standard Chrome user-agent string.

These measures help with sites that only perform basic automation checks. They are not sufficient for sites with enterprise bot protection (Cloudflare, Akamai, PerimeterX, DataDome, etc.).

### Choosing an automation approach

Your choice of [automation approach](./automation-approaches) is the biggest factor in detection risk:

| Approach                                     | Detection risk   | Why                                                                             |
| -------------------------------------------- | ---------------- | ------------------------------------------------------------------------------- |
| Passive interception (`page.on('response')`) | **Low**          | Makes no extra requests — just observes normal browser traffic                  |
| Regular Playwright                           | **Moderate**     | Browser fingerprint and behavioral patterns can be analyzed                     |
| In-browser fetch (`pageRequest()`)           | **Low–moderate** | Uses the real browser's network stack, but some sites monitor fetch call stacks |
| Direct HTTP from Node.js                     | **Very high**    | Completely different TLS/HTTP fingerprint from a real browser                   |

When possible, prefer passive interception or in-browser fetch over direct HTTP requests.

### Signs your automation is being detected

Watch for these during development:

- **Challenge pages** — "Checking your browser..." interstitials, CAPTCHAs, or blank pages before content loads
- **HTTP 403 / Access Denied** — Requests that work in a normal browser but fail in automation
- **Degraded results** — Fewer items returned, missing data, or slower responses compared to manual browsing
- **Account flags** — Login failures, security alerts, or account lockouts when using authenticated sessions

### Tips for reducing detection risk

- **Use headed mode** — Headless browsers have detectable differences in rendering and API behavior. Libretto launches in headed mode by default.
- **Add realistic delays** — Use `page.type()` with variable delays instead of `page.fill()`. Pause between actions like a human would.
- **Reuse authenticated profiles** — Use `libretto save` to persist session state. Logging in every run is more suspicious than reusing a saved session.
- **Avoid cloud IPs** — Requests from AWS/GCP/Azure IP ranges face higher scrutiny. Run from a residential IP when possible.
- **Check for fetch patching** — Before using `pageRequest()`, open the browser console and run `window.fetch.toString()`. If it returns actual JavaScript (not `"[native code]"`), the site monitors fetch calls — prefer passive interception instead.
